Add a consistent "forgot password / forgot PIN" reset flow everywhere a
credential is entered. Use ONE shared reset mechanism (via the existing
Resend email setup), applied to all three credential types:
  (a) main/PM account password
  (b) portal member password
  (c) sign-off PIN

1. SHARED RESET BACKBONE:
   - "Forgot password?" / "Forgot PIN?" link on every relevant entry
     screen (PM login, portal login, set-password, PIN prompt, and in
     settings).
   - Generates a secure, single-use, hashed, time-limited token (30–60
     min expiry) stored server-side; the email contains the only copy of
     the link/code.
   - Sends a reset email via Resend (from the verified
     mail.sitesort.co.uk domain) with a clear "reset your password/PIN"
     button + fallback link, and a short plain-text version.
   - Link → a reset page where they set a new password/PIN (same
     validation rules as normal: password strength; PIN hashed, min
     length). Token consumed on use; expired/used tokens show a friendly
     "this link has expired, request a new one" page.

2. SECURITY REQUIREMENTS (must-haves):
   - IDENTICAL response whether or not the email exists ("if that address
     has an account, we've sent a reset link") — never reveal which
     emails are registered.
   - Rate-limit reset requests per email/IP (e.g. a few per hour).
   - On successful reset, INVALIDATE existing sessions for that account
     (and for a PIN reset, just re-set the PIN) so a compromised account
     can't stay logged in elsewhere.
   - Tokens hashed at rest; never logged; one-time use.
   - Reset events recorded in the activity log (requested, completed).

3. PIN-SPECIFIC: a PIN reset re-verifies the person is logged in to their
   own account OR uses the same email-token route if they're locked out
   entirely; new PIN stored hashed. Resets logged.

QUALITY: OpenAPI + codegen, Zod, migration for reset tokens, all
verification server-side, typecheck + check:layout at 360px (reset pages
are used on phones).

VERIFY — test each of the three end to end and report pass/fail:
- request a password reset for a REAL account → email arrives → link sets
  a new password → can log in with it → old password no longer works →
  existing sessions were invalidated
- request for a NON-EXISTENT email → same generic response, NO email, no
  error revealing it doesn't exist
- reuse a spent link → rejected; use an expired link → friendly expiry
  page
- portal member password reset → same flow works on mobile
- sign-off PIN reset → new PIN works, old PIN rejected, event logged
- confirm rate-limiting blocks rapid repeated requests
Report the token model and confirm tokens are hashed + single-use.
